# 编写 Skills

## 流程

1. **收集需求** - 询问用户：
   - 该 Skill 覆盖什么任务/领域？
   - 它应处理哪些具体用例？
   - 它需要可执行脚本，还是只需说明即可？
   - 是否需要包含参考资料？

2. **起草 Skill** - 创建：
   - SKILL.md，包含简洁的说明
   - 如果内容超过 500 行，添加额外的参考文件
   - 如果需要确定性操作，添加实用脚本

3. **与用户一起审阅** - 展示草稿并询问：
   - 是否覆盖了你的用例？
   - 是否有遗漏或不清楚的地方？
   - 某些部分是否需要更详细/更简略？

## Skill 结构

```
skill-name/
├── SKILL.md           # 主说明文件（必需）
├── REFERENCE.md       # 详细文档（如需要）
├── EXAMPLES.md        # 使用示例（如需要）
└── scripts/           # 实用脚本（如需要）
    └── helper.js
```

## SKILL.md 模板

```md
---
name: skill-name
description: 能力简要说明。当 [具体触发条件] 时使用。
---

# Skill 名称

## 快速开始

[最小可用示例]

## 工作流程

[复杂任务的分步流程及检查清单]

## 高级功能

[链接到单独文件：参见 [REFERENCE.md](REFERENCE.md)]
```

## 描述要求

描述是你的智能体在决定加载哪个 Skill 时**唯一能看到的内容**。它会与所有其他已安装 Skill 一起出现在系统提示中。智能体阅读这些描述，并根据用户请求选择相关的 Skill。

**目标**：让你的智能体仅通过描述就能知道：

1. 该 Skill 提供什么能力
2. 何时/为何触发它（具体关键词、上下文、文件类型）

**格式**：

- 最多 1024 个字符
- 使用第三人称
- 第一句：它做什么
- 第二句：“当 [具体触发条件] 时使用”

**好的示例**：

```
从 PDF 文件中提取文本和表格、填写表单、合并文档。当处理 PDF 文件或用户提到 PDF、表单、文档提取时使用。
```

**差的示例**：

```
帮助处理文档。
```

差的示例让你的智能体无法将它与其他文档类 Skill 区分开。

## 何时添加脚本

在以下情况添加实用脚本：

- 操作是确定性的（验证、格式化）
- 同一段代码会被重复生成
- 错误需要显式处理

与生成代码相比，脚本可以节省 token 并提高可靠性。

## 何时拆分文件

在以下情况拆分为单独文件：

- SKILL.md 超过 100 行
- 内容涉及明显不同的领域（如金融与销售 schema）
- 高级功能很少被用到

## 审阅检查清单

起草完成后，请确认：

- [ ] 描述包含触发条件（“当...时使用”）
- [ ] SKILL.md 在 100 行以内
- [ ] 没有时效性信息
- [ ] 术语一致
- [ ] 包含具体示例
- [ ] 引用仅深一级
