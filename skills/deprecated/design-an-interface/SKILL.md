# 设计接口（Design an Interface）

基于《A Philosophy of Software Design》中的 "Design It Twice"：你的第一个想法通常不是最好的。先生成多个截然不同的设计，然后进行比较。

## 工作流程

### 1. 收集需求

在设计之前，先搞清楚：

- [ ] 这个模块要解决什么问题？
- [ ] 调用方是谁？（其他模块、外部用户、测试）
- [ ] 关键操作有哪些？
- [ ] 有什么约束？（性能、兼容性、现有模式）
- [ ] 哪些应该封装在内部，哪些应该暴露？

追问："这个模块需要做什么？谁会使用它？"

### 2. 生成设计方案（并行子智能体）

使用 Task 工具同时启动 3 个以上的子智能体。每个智能体必须产出一种**截然不同**的方案。

```
每个子智能体的提示模板：

为以下模块设计接口：[模块描述]

需求：[已收集的需求]

本次设计的约束：[为每个智能体分配不同的约束]
- Agent 1: "最小化方法数量——最多 1-3 个方法"
- Agent 2: "最大化灵活性——支持尽可能多的使用场景"
- Agent 3: "为最常见场景优化"
- Agent 4: "从[特定范式/库]中汲取灵感"

输出格式：
1. 接口签名（类型/方法）
2. 使用示例（调用方如何使用它）
3. 该设计在内部隐藏了什么
4. 该方案的权衡
```

### 3. 展示设计方案

逐一展示每个设计，包含：

1. **接口签名** —— 类型、方法、参数
2. **使用示例** —— 调用方在实践中如何使用它
3. **隐藏了什么** —— 哪些复杂性被保留在内部

按顺序展示设计方案，让用户在比较之前先理解每一种方案。

### 4. 比较设计方案

展示完所有方案后，从以下维度进行比较：

- **接口简洁性**：方法更少、参数更简单
- **通用 vs 专用**：灵活性 vs 聚焦性
- **实现效率**：接口形态是否允许高效的内部实现？
- **深度**：小接口隐藏大量复杂性（好）vs 大接口加薄实现（不好）
- **正确使用容易度** vs **误用容易度**

用文字讨论权衡，而不是列表。重点指出方案之间分歧最大的地方。

### 5. 综合

最好的设计通常结合了多个方案的优点。追问：

- "哪种设计最符合你的主要使用场景？"
- "其他设计中有哪些元素值得融入？"

## 评估标准

来自《A Philosophy of Software Design》：

**接口简洁性**：方法越少、参数越简单 = 越容易学习并正确使用。

**通用性**：能在不修改的情况下应对未来的使用场景。但要警惕过度泛化。

**实现效率**：接口形态是否允许高效的实现？还是迫使内部实现变得别扭？

**深度**：小接口隐藏大量复杂性 = 深模块（好）。大接口加薄实现 = 浅模块（应避免）。

## 反模式

- 不要让子智能体产出相似的方案——必须保证差异显著
- 不要跳过比较环节——价值正来自于对比
- 不要实现——这纯粹是关于接口形态
- 不要基于实现工作量来评估
